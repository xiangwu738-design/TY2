## 1. 前置环境（P0，云端 VM 自行安装）

1. **Godot .NET 版**：`Godot_v4.7-stable_win64_dotnet.exe`（普通版跑不了 C#）。下载到 `~/godot`。
2. **.NET 8 SDK**：`apt` 或官方脚本装 `dotnet-sdk-8.0`。`dotnet --version` 可用。
3. **git**：`apt install git`。
4. **tmux**：常驻用。
5. clone 工程（含本 docs/）。若用户尚未推 git，先在本地 `git init` 推 GitHub 私有仓（问用户要仓库地址）。
6. 验证：`godot --headless --path . --quit-after 10` 无错；`dotnet build` 可编译 C# 工程。

## 2. 工程结构（C# Godot）

```
NEO/
  project.godot           (Godot .NET, gl_compatibility)
  *.csproj / *.sln        (C# 工程)
  src/Core/               纯逻辑内核 C#（不依赖 Godot Node，可单测、可克隆预演、确定性）
    Enums.cs Card.cs CardDef.cs Character.cs Enemy.cs Timeline.cs GameState.cs
    Enchantment.cs Shield.cs  (附魔/护盾)
  src/Views/              表现层（GameView 等，C# Control）
    GameView.cs PortraitController.cs ...
  src/Net/                联机层（C#，ENetMultiplayerPeer + RPC）
    NetController.cs
  src/Roguelike/          外壳（地图/商店/遗物/休息/事件）
  src/Data/               模板数据（角色模板/卡池占位/敌人编排占位）
  tests/                  C# 单测（或 Godot headless 自检场景）
  docs/                   企划书/问题清单/本规格/待用户填参数
  tools/slack-bridge/     (已有)
```

**分层铁律**：Core 不 `using Godot`（用纯 C# 类型），可被单测与 `GameState.Clone()` 预演；Views/Net 只读写 Core。确定性：除开局带种子洗牌外，结算零 RNG。

---

## 3. 架构

- **Core（C#）**：`GameState` 持有角色/敌人/时间轴/指针/事件流；`Preview(action)` 在 `Clone()` 上跑给出「松手前」完整后果；`PlayCard/UsePrep/Skip` 返回事件流。
- **Views**：`GameView`（每进程一个）订阅 Core 事件流渲染战场/时间轴/手牌/预览/日志；`PortraitController` 播立绘状态机。
- **Net**：主机权威 + 动作同步。客户端发 action→主机执行→广播 action→各端同种子确定性重放。中途加入：主机发 seed + 历史动作重放。
- **事件流**：Core 产出 `List<GameEvent>`；Views 据事件播放动画/更新 UI；Net 同步 action（不同步状态）。

## 4. 子系统规格

### 4.1 时间轴与占位（P1）
- 单层格子，指针逐格推进。占位=推进格数=时间成本。
- 逐格结算：进格→①触发敌人节点→②检查持续态(护盾)→③若是占位终点则结算牌效果。
- 高占位=效果迟=沿途更易先挨打（关键语义，必须有预览佐证）。

### 4.2 出牌模型（P1，重做）
- **整备牌**：每角色一张固定整备牌，**打出后回手**（不从手牌消失/不进弃牌堆），效果=抽牌（抽几张/是否弃牌用模板参数，见 §7）。占位消耗时间。
- **其它牌**：打出→进弃牌堆，**不抽牌**。手牌靠整备补给。
- 抽牌堆空→弃牌堆整叠放回（带种子洗牌，输入端随机；联机需同步种子）。
- 空过：推进1格，无效果。
- ⚠️ 废弃 MVP 的「打一张补一张（_draw_after_play）」。

### 4.3 位置系统（P1）
- 位置 1..N（N=玩家数，**缩减人数**），1=前线。出攻击牌→移到1位，身前的人后移（输出即暴露）。
- 阵亡收缩：死人后身后的人向前补位，保持 1..N 连续。
- 敌人三型：斩=位1、突=位2、打=全体（若 N<2，突可落空）。
- 预读：被瞄准位置+其上玩家高亮；预打出预览（占位用各角色色）。

### 4.4 附魔挂具体牌（P1，重做）
- 引入 **Card 实例层**：手牌/弃牌堆/抽牌堆里的每张牌是独立实例，可挂 `Enchantment`。
- 附魔作用域：可挂到「某张具体牌」「弃牌堆所有牌」「抽牌堆所有牌」等；玩家靠打牌顺序兑现（找秩序）。
- 已有附魔类型保留：力量（攻击+mag）、易伤（受击+mag ×N次）、蓄力（敌力+1）。预留扩展：减伤、占位改动、抽牌扰动、条件增伤。
- **标记=显示层**：敌人/角色身上的标记只反映附魔状态，不存独立数据。
- ⚠️ MVP 的 holder 简化废弃，改为挂具体牌实例。

### 4.5 护盾（P1）
- **两型**：固定吸收量（挡到耗尽）、次数型（挡N次）。不同护盾牌不同机制。
- 绑**守护关系**：为某位铺盾，被守护者离位则盾消失。
- 护盾是时间轴上的「关系持续态」，在逐格结算第②步检查并吸收。

### 4.6 血量上限（P2）
- 个人血，非共享。每场开局回满当前血。
- 战斗结算按受伤程度下调**血量上限**（跨场长期损伤）；上限只能靠**休息**恢复。
- 被击倒→本场临时 debuff（弱而不废），本场结束清除；长期代价由上限下调独担。
- 上限一路被压→重伤→天堂线（叙事死亡阶梯，占位）。
- 速率参数（掉多快/回多易）用模板占位，登记到 §7。

### 4.7 拼刀/破绽（扩展位，不实现）
- **不写拼刀逻辑**，但留扩展位：
  - 节点类型枚举预留 `VULN_NODE`/破绽节点。
  - 攻击牌经过敌人节点处留 hook（`OnAttackPassNode`）。
  - 破绽节点作为时间轴节点类型预留（伤害加成占位）。

### 4.8 战斗布局与立绘状态机（P1 表现层 / P5 占位美术）
- side-view 回合制：角色一侧（位 N..1 排开）、敌人另一侧；时间轴+牌区在下。
- `PortraitController`（每角色/敌人一个）：
  - 状态：`Idle`（循环帧）、`Skill`（出招帧，播完回 Idle）、预留 `Hit`/`Down`。
  - 由 Core 事件驱动：角色牌效果落地→`Skill`；敌人节点触发→敌人 `Skill`。
  - 动画异步、不阻塞结算（结算即时完成）。
- 占位：色块/简单帧循环；统一接口供换立绘。

### 4.9 roguelike 外壳（P3）
- 地图：分层选路（尖塔式），节点类型 战斗/精英/商店/休息/事件/Boss。层数用模板参数。
- 战后：加牌（受控，质量保底）、移牌。
- 商店：货币(金币占位)、商品(牌/遗物/移牌/恢复上限)。
- 休息：恢复血量上限（具体量用模板参数）。
- 遗物：挂占位/时间轴机制；数据驱动，留注册接口。
- 卡组偏小（12–18），构筑主轴=占位/时间。

### 4.10 角色模板（P4）
- 4 定位模板：输出 / 布防 / 控制 / 治疗（方向占位，具体由用户填）。
- 每角色：专属卡池（占位几张）、专属整备牌（模板：占2抽2不弃、回手）、专属颜色。
- 卡池/整备具体内容登记到 §7，用户后填。

### 4.11 联机（P2）
- C# `ENetMultiplayerPeer`，主机权威 + 动作同步。
- **缩减人数**：N 玩家 = N 角色（位 N..1）。主机=玩家之一（c0）或专用服（见下）。
- 中途加入：主机发 seed + 历史动作，客户端重放追平。
- 留 dedicated-server 模式扩展位（无画面纯仲裁），供日后云端常驻。
- 单人模式角色数：**待用户确认**（1 vs 4）——见 §7，先用「单人=4 角色（控全部）」模板，用户可改。

## 5. 阶段与自验

| 阶段 | 产出 | 自验 |
|---|---|---|
| P0 | 环境 + C# 工程骨架 | `dotnet build` 通过；godot headless 起空场景无错 |
| P1 | Core（时间轴/出牌模型重做/位置/附魔挂牌/护盾）+ 立绘状态机占位 | C# 单测：逐格结算、整备回手、附魔挂牌生效、护盾吸收、阵亡收缩、确定性预演 |
| P2 | 血量上限 + 联机(缩减人数) | 单测上限下调；双进程动作同步指针一致 |
| P3 | roguelike 外壳 | 一局能跑通 地图→战斗→加牌→商店→休息→Boss |
| P4 | 角色模板 + 专属卡池占位 | 4 角色各能玩 |
| P5 | 占位美术 + UX + 导出 | 截图无重叠；exe 导出冒烟 |

每阶段：`git commit` + Slack 汇报。

## 6. 占位与扩展位约定

- 占位：色块/文字/简单帧。数值用合理默认并登记 §7。
- 扩展位：拼刀、dedicated-server、更多附魔/节点/遗物——留接口与枚举，不实现。
- 凡是「具体内容」（卡牌数值、角色立绘、关卡编排、Boss、美术）一律**不自行编造**，用模板占位 + 登记 §7。
