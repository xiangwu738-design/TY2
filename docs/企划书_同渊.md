## 0. 一句话定位

四人协作（联机）+ 单人的**卡牌战术 roguelike**。用「时间轴 + 占位」取代费用/能量，用「完全信息预读 + 被驯服的随机」取代事后掷骰，主打**可决策的协作战术**与**低挫败的温暖小队**体验。

## 2. 设计理念

1. **完全可决策**（由「完全信息」松绑而来）：玩家做决策的那一刻，信息足够算清后果。随机**只允许存在于输入端**（抽到什么牌、关卡出什么题），出牌后的结果不靠事后掷骰。
2. **被驯服的随机**：随机发生在输入端，玩家用打牌顺序去驾驭；决策结果始终确定可算。是手艺，不是骰子。
3. **低挫败、温暖小队协作**：守住不滚雪球的地板，协作本身就是复杂度的分摊器。
4. **复杂度警惕**（来自规则 §10）：系统已相当庞大，且每解一题几乎都靠「再加一个机制」。每加一项时主动问：「它配得上我要的简单吗？」**总复杂度的增长本身，是比任何单点都更该警惕的趋势。**

---

## 3. 核心循环

### 局内循环
看时间轴预读 → 选角色出牌（占位推进）→ 沿途敌人节点先触发 → 牌效果在占位末尾落地 → 重整手牌 → 直到敌人被击败。

### 局外循环（roguelike 外壳）
地图选路 → 战后加牌/移牌 → 商店 → 遗物 → 下一场 → Boss → 通关/失败。

## 4. 核心机制详述

### 4.1 时间轴与占位 【已验·已实现】
- **时间轴 = 一串格子**，指针从左向右逐格推进。废除「间隙」层，一切都挂在格子上。
- **占位 = 推进的格数 = 时间成本**（取代费用/能量）。占位 0 只能设计师写死，不可由减费动态产生。
- **逐格结算顺序**（指针每新进入一格）：① 触发该格敌人事件点 → ② 检查持续态（护盾）→ ③ 若是本牌占位终点则结算本牌效果。
- **关键语义**：牌效果在占位末尾才落下，沿途敌人节点先触发 ⇒ **打牌时敌人可能抢在你效果生效前先出手**。占位越高、效果越迟、途中越可能先挨打。
- 格子上能放：事件点（敌人行动节点、破绽节点）、关系持续态（护盾）。

### 4.2 出牌与手牌 【已实现·已修订】
> ⚠️ 此处与原规则 §2 有重大分歧，需确认（见 §11-Q1）。

- **打一张补一张**：打出 → 进弃牌堆 → 自动抽 1 补回。手牌恒定（起手 5）。
- **无能量限制、无整备**：出牌唯一成本是占位/时间。原 §2 的「出牌不抽牌 + 固定整备（弃1抽2）」**已废弃**。
- **抽牌堆空 → 弃牌堆整叠放回抽牌堆**（确定性循环；正式版可换带种子洗牌）。
- **空过**：只推进 1 格，永远可用（走位/等待）。

### 4.3 位置系统 【已验·已实现】
- 四位，从左到右 4 / 3 / 2 / 1，1 位最前线。四名玩家各有专属颜色。
- **出攻击牌 → 该玩家移到 1 位**，其身前的人依次后移（**输出即暴露**）。
- **敌人攻击三型**：斩击=打 1 位、突刺=打 2 位、打击=打全体。被瞄准位置 + 其上玩家预先高亮。
- **阵亡收缩**（MVP 新增）：有人倒下时，身后的人向前补位，保持 1..N 连续、前线锚定 1。例：4 人→死 1→位 3,2,1。
- **预打出预览**：拖牌未松手时全员可见占位预览（用其颜色）、攻击牌「将移到 1 位」预览。松手才结算。
- **数值预览**：所有伤害预览实时反映当前修正（附魔、敌我状态），所见即实际生效数字。

### 4.4 附魔与标记 【基础已实现】
> 规则 §4。所有 buff 抽象为「附魔」，载体是**牌**这个实体，与「以牌为核心货币」统一。

- **机制层（附魔）**——MVP 已实现 3 种：
  - **力量**（玩家侧，永久）：该角色攻击牌 +mag。牌「聚力」施加。
  - **易伤**（敌人侧，次数型）：对敌攻击 +mag，按次数扣减，归零移除。牌「破甲」施加。
  - **蓄力**（敌人侧，经编排触发）：敌人节点「蓄」触发，敌力 +1。体现敌我对称。
- **显示层（标记）**：附魔是机制载体（管怎么生效、作用在谁）；敌人/角色身上的标记是显示层（汇总附魔状态成可见图标）。**标记不持有独立数据，只反映附魔这一个真相源**——层数/时效由所联动附魔决定。两层分工、不重复、不互相存数据。✅ 已按此实现。
- **【待实现·开放】**：附魔挂在**具体牌**（手牌/弃牌堆/抽牌堆）的「在混沌中找秩序、主动控牌」玩法——这是 §4 的精髓，MVP 用 holder 简化了。
- **【待实现】**：更多附魔类型（减伤、占位改动、抽牌扰动、条件增伤等）。

### 4.5 防御体系 【部分已实现】
基础防御三条腿：
- **走位规避**：预读敌人打哪个位，别站那。【已实现：挪位牌 / 换人上前线】
- **护盾**：他人布防，绑位置/守护关系。【待实现】
- **治疗**：善后，接住漏掉的伤害。【已实现：调息/包扎】

**拼刀流**（进阶流派，非基础）【待实现】：
- 攻击牌（占位 ≥ 2）经过敌人节点 → 比伤害。赢 → 完全无效化该次攻击 + 生成**破绽节点**；输 → 全额挨打、不抵扣（全有全无）。占位 0/1 弱牌不能拼。Boss 大招等标为**不可拼**。
- **破绽节点 = 本作的「暴击」**（条件触发的确定爆发，零随机）：拼赢后生成，任何队友攻击牌经过 → 吃伤害加成 →「一人拼赢开窗、众人扑上」的协作连携。
- 原则：匹配从「系统替所有人保证」降为「这个流派主动构筑去够」。

### 4.6 血量体系 【部分待实现】
> 规则 §7。

- **个人血**（非共享）。【已实现】
- **每场开局回满当前血** → 不带残血进场，守住低挫败地板。【已实现】
- **战斗结算时按受伤程度下调「血量上限」** → 提供跨场、有分量的长期损伤资源；**上限只能靠休息恢复**（给「休整/不参战」具体数值意义）。【待实现】
- **被击倒 → 加 debuff**：仅本场内临时削弱（必须「弱而不废」，不致坐牢拖累），本场结束清除；长期代价由「上限下调」独自承担，两者不叠。【部分待实现：倒下debuff】
- 上限一路被压低 → 通向**重伤 → 天堂线**（叙事死亡阶梯）。
- **命门**：「上限掉多快 vs 休息回多易」这对速率，是本机制成败的全部所在 —— 调到「每次掉上限都肉疼、又不会一路单向崩盘」的甜区。**只能原型调。**

### 4.7 构筑（roguelike 外壳）【待实现】
> 规则 §8。

- 尖塔式结构：地图、战后加牌、商店、移牌、遗物。单多人共用。
- **成型卡组偏小（约 15 张，12–18 区间）**：因「打牌补牌快循环 + 完全信息要可算 + 四人出牌权稀缺」，机制天生奖励「小而精」。起手单薄（~12）、成型长到 15–18。
- **获取牌偏受控**（近 Chrono Ark 而非尖塔慷慨），配「质量保底 + 移牌顺畅」软化挫败。
- **构筑主轴是占位/时间**（而非能量曲线）。遗物挂占位/时间轴机制（守占位下限等）。

### 4.8 四人协作与联机 【架构已实现】
> 规则 §9。

- **共享时间轴**：四人共用一条轴、一个指针，出牌权是全队共担的稀缺资源。
- 故四人牌走「少而重 + 协作增益/连携」（我的牌让你更狠），把「出得少」变成「必须配合」。
- **认知分工**：四人不必各盯全部信息，可让不同角色主理不同维度（输出/布防/读意图），让协作顺便成为复杂度的分摊器。
- **联机**：主机权威 + 同步动作（非状态）；ENet；4 独立进程/窗口。
  - 同种子 + 确定性重放：客户端发动作给主机 → 主机权威执行 → 广播动作 → 各端重放。
  - 中途加入：主机发 seed + 历史动作，客户端重放追平。
  - localhost 多窗口 → 局域网 →（未来）公网中继。
- **【待办】**：端到端实测联机回路（双进程对齐指针）；<4 人时代控闲置角色方案。

## 5. 角色设计方向 【开放】

- 固定四角色，各有专属颜色与定位（甲乙丙丁为占位名）。
- **【开放】** 差异化程度（见 §11-Q4）：
  - 选项 A：四人共用同一套牌（MVP 现状，最简）。
  - 选项 B：各有专属卡池 + 专属定位（输出/布防/读意图/治疗）。
  - 选项 C：专属卡池 + 专属「整备/运转」差异。
- 认知分工建议：让四人分别主理不同维度，使协作成为复杂度分摊器。

## 6. 敌人与关卡 【方向已定·待设计】

- **敌人 = 关卡**：没有牌堆循环，它的「牌」是设计师预先编排、铺在时间轴上的序列（像谱子），玩家走到触发。
- **节点布局是关卡语言**：疏密/前后编码敌人性格与攻击节奏（快攻=密且靠前；成长=疏且靠后）。**平衡主旋钮在节点布局，而非牌数值**（类《陷阵之志》）。
- **分支，而非随机**：优秀关卡对玩家行动做分支——敌人会因你的站位/血量/行为切换行为，但**切换条件明牌、可预读**。给「活对手、不背板」的全部好处，又不破坏可决策性。
- **少量输入端随机**（敌人组合、序列变体、初始站位）用于重玩性；但**不在玩家决策后用不可见骰子改变结果**（尤其警惕「伤害值小幅浮动」这种最廉价、却最伤害精确计算的随机）。
- **Boss**：大招等标为**不可拼**。
- 节点类型（已有）：斩击/突刺/打击/蓄力。**【待设计】** 更多节点类型与 Boss 编排。

## 7. 技术架构

- **引擎**：Godot 4.7，GDScript（内核未来可迁 C# 强类型）。
- **严格分层**：
  - **纯逻辑内核**（`scripts/core/`，RefCounted）：确定性、可单测、可克隆预演（`preview_action` 在克隆状态上跑一遍给出「松手前」完整后果）。不依赖任何 Godot 节点。
  - **表现层**（`scripts/ui/`，`GameView`）：只渲染 + 转发动作，不持规则逻辑。每进程一个 GameView。
  - **联机层**（`main.gd`）：主机权威 + 动作同步。
- **确定性**：除开局带种子洗牌外，结算全程无 RNG，可重放——这是「完全信息预读」和「动作同步联机」的共同基石。
- **目录结构**（已建立）：
  ```
  scripts/core/   enums card card_defs character enemy game_state
  scripts/ui/     game_view main(控制器+联机)
  scripts/test_kernel.gd   headless 内核自检
  main.tscn  project.godot  export_presets.cfg
  docs/企划书_同渊.md
  ```
- **未来扩展点**：内核迁 C#；联机公网中继；附魔挂具体牌（需 Card 实例层）。

## 10. 复杂度账本与风险（诚实备注）

- 系统已相当庞大，且最近每解决一个设计问题，几乎都是用「再加一个机制」来解（附魔、拼刀流、血量上限、阵亡收缩…）。单个都自洽，但**总复杂度的增长本身，是比任何单点都更该警惕的趋势**。
- **尚未经原型的关键手感点**（来自规则 §10，必须做出来玩）：
  1. 血量「掉上限/休息恢复」速率；
  2. 位置系统下的四人协作是否好玩；
  3. 破绽节点的位置与时效；
  4. 附魔制在实战中的认知负担。
- **风险**：联机动作同步在跨平台/弱网下的鲁棒性（目前仅 Windows 同版本验证）；附魔挂具体牌后的认知负荷可能过高。

## 11. 开放问题（需你拍板）

> 这些是正式版前必须定的岔路。括号为 Claude 的倾向，仅供参考。

- **Q1 出牌模型**：现为「打一张补一张、无能量、无整备」（你 MVP 中修订）。是否保留？还是回退原 §2「出牌不抽牌+整备（弃1抽2）」？
  → 倾向：保留「打一张补一张」。它让唯一成本=时间，与「占位/时间主轴」最自洽；稀缺感来自时间轴推进（触发敌人），而非手牌耗竭。但会削弱「手牌是消耗资源」的紧张感——需你确认是否接受。
- **Q2 闲置角色**：联机 <4 人时，未连接角色闲置。是否允许主机代控闲置角色？还是接受闲置？
  → 倾向：允许主机代控未被人认领的角色（点头像切换），便于 2–3 人也能玩。
- **Q3 附魔载体**：holder 简化（现状）vs 挂具体牌（手牌/弃牌堆/抽牌堆的「找秩序」玩法）。
  → 倾向：正式版做「挂具体牌」——这是 §4 的精髓与差异化卖点，但需控制认知负荷（见 Q4 风险）。
- **Q4 角色差异化**：共用牌池 vs 专属卡池+定位 vs 专属卡池+运转差异。
  → 倾向：专属卡池+定位（输出/布防/读意图/治疗），配合认知分工。
- **Q5 血量上限长期损伤**：是否做？它给跨场分量，但与「低挫败」有张力。
  → 倾向：做，但速率温柔，且休息恢复要顺畅。
- **Q6 拼刀流**：是否纳入正式版？原规则降为「进阶流派」。
  → 倾向：作为可选流派纳入，不强制全员。
- **Q7 美术风格**：待定。
- **Q8 单人模式定位**：单人=控制全部 4 角色（现状）vs 单人=纯 1 角色简化版？
  → 倾向：单人=控制全部 4 角色（与多人共用规则，§0 定位）。

## 15. 战斗布局与立绘/状态机（v0.2 新增）

> 老式回合制 RPG 布局，重立绘与状态机。

- **布局**：经典 side-view 回合制——四角色在画面一侧（按站位 4→1 排开），敌人在另一侧。时间轴/行动条与牌区在下方。
- **角色立绘状态机**：每个角色有
  - **待机状态**：循环播放数张待机立绘（呼吸/眨眼 loop）。
  - **技能状态**：伴随时间轴推进、该角色出牌（牌效果落地）时切入，播放技能立绘，播完回到待机。
  - （预留：受击、倒下状态。）
- **敌人同理**：待机循环；敌人节点触发（攻击）时切入技能/攻击立绘，播完回待机。
- **驱动**：状态切换由内核事件驱动（出牌落地、节点触发），表现层订阅内核事件流播放对应动画。内核不依赖动画（确定性），动画只是事件的可视化。
- **占位**：用色块/简单帧占位立绘，留统一接口（`PortraitController`：set_state(idle/skill/hit/down)、帧序列）供你之后替换成正式立绘。
- **实现**：Godot `AnimationPlayer`/`AnimatedSprite2D` + 状态机；每角色/敌人一个 `PortraitController` 节点。
- 注意：立绘播放不阻塞内核结算（结算即时完成，动画异步播放或截短），避免拖慢节奏与破坏确定性。
